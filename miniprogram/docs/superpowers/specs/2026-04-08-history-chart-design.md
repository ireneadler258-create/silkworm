# 历史页面曲线图优化设计

## 目标
在不重构整体项目架构的前提下，优化 `src/pages/history/index.vue` 的历史曲线图体验，解决以下问题：

1. 1h 范围下图表“全白 / 看不到内容”
2. 历史页有时加载较慢
3. 页面整体视觉不够协调
4. 保持当前项目的 uni-app + Vue 3 + TypeScript 风格与现有页面一致

## 当前实现概览

### 相关文件
- `src/pages/history/index.vue`
  - 历史页面入口
  - 时间范围切换、指标切换、趋势文字、统计摘要、告警记录
- `src/components/LineChart.vue`
  - 自绘 canvas 折线图
  - 可横向滚动、点选、阈值线、网格与空态文案
- `src/store/device.ts`
  - 历史数据本地缓存
  - 云端历史同步 `fetchCloudHistory()`
  - 本地读取 `getHistory(rangeHours)`
  - 历史采样 `recordHistoryPoint()`

## 问题分析

### 1. 1h 视图“全白 / 看不到内容”
根因不是单一 bug，更像是多个因素叠加：

1. `LineChart.vue` 里图表区域底色较浅、线条与网格透明度偏低，空态文案也是偏白，导致低数据量时视觉上像整块空白。
2. `history/index.vue` 在 `arr.length < 2` 时会人为补两个点，这能避免完全空数组，但不等于“有可读趋势图”。1h 场景下如果最近采样点稀疏，界面容易进入“技术上有数据、视觉上像没数据”的状态。
3. 当前状态缺少明确区分：`loading / empty / low-data / ready` 被混在一起，只要 `values.length >= 2` 就按正常图表渲染，不利于用户理解。

### 2. 页面有时加载慢
主要压力来自 `device.ts`：

1. 初始化时直接调用 `fetchCloudHistory()`。
2. 云端同步按多个属性循环请求阿里云历史数据，并逐页合并。
3. 历史页渲染同时还要做计算属性、降采样、canvas 重绘、点击热区生成。
4. 当前没有对历史页可见性与“云同步完成”做解耦，导致用户感知到“页面慢”。

### 3. 页面视觉不够协调
当前已经有玻璃卡片与渐变背景，但历史页还存在：

1. 图表卡片信息层级不够清晰
2. 范围 tab、指标 tab、趋势条、统计区样式风格稍微分散
3. 图表文字与背景对比不足，精致感不够稳定

## 方案选择

### 方案 A：最小修复
仅修 1h 可见性和少量样式问题。

- 优点：改动最小
- 缺点：慢加载问题改善有限

### 方案 B：平衡优化（采用）
同时处理图表状态、性能感知和视觉统一，但不重做架构。

- 优点：最符合当前诉求
- 缺点：改动中等，需要同时改页面、图表和 store

### 方案 C：重做分析页
重构为更完整的数据分析体验。

- 优点：上限最高
- 缺点：超出“优化页面”的范围

## 最终设计（采用方案 B）

### 一、历史页状态分层
在 `src/pages/history/index.vue` 中引入明确的视图状态：

- `loading`：历史数据正在准备
- `empty`：当前范围没有有效数据
- `low-data`：只有很少的数据点，不适合展示完整趋势，但需要显示当前值与基础信息
- `ready`：正常渲染图表

这样可以把“无数据”“数据少”“正常数据”分开处理，避免 1h 看起来像故障。

### 二、1h 场景优化策略
#### 目标
确保 1h 场景下永远不会出现“全白、不知道发生了什么”的体验。

#### 处理方式
1. 当真实点位不足以形成可信趋势时，不再只依赖补点后的伪趋势。
2. 展示更明确的占位内容：
   - 当前指标值
   - 最近更新时间
   - 提示文案（如：当前 1h 数据较少，已显示最近采样）
3. 图表组件支持“低数据模式”：
   - 保留简化折线或关键点
   - 提高主线与关键点对比度
   - 背景不再过亮
4. 空态文案和低数据提示使用深色文字与轻背景胶囊，不再与图表背景混在一起。

### 三、图表组件优化
在 `src/components/LineChart.vue` 中调整：

#### 1. 视觉层
- 提升主折线颜色饱和度和不透明度
- 降低背景蒙层发灰/发白感
- 提升坐标文字与网格可读性
- 优化阈值线与异常点层次
- 选中点 tooltip 更精致、更稳定

#### 2. 交互层
- 修正 `clickableDots` 与 `onDotTap()` 的索引步长不一致问题，确保点击点位映射准确
- 在数据量大时继续限制可点击点数量，减少热区节点数量
- 降低不必要重绘频率

#### 3. 状态层
为组件增加对以下状态的兼容：
- `loading`
- `empty`
- `low-data`

组件在这些状态下渲染不同视觉，不再只依据 `values.length < 2` 判断。

### 四、数据获取与性能优化
在 `src/store/device.ts` 中保持现有结构，但优化用户感知：

#### 1. 首屏优先本地数据
历史页优先展示本地 `history`，即使云同步尚未完成，也先让页面可用。

#### 2. 弱化云同步对页面的阻塞感
不要求页面等待 `fetchCloudHistory()` 完成后才可见，而是：
- 本地先出结果
- 云同步完成后再渐进刷新

#### 3. 避免无效计算放大
- 仅在必要时生成图表点位、统计信息与热区
- 控制大区间时的点密度
- 保持现有 `downsample` 思路，但让显示状态优先于“强行画图”

## 视觉设计原则
保持项目现有的玻璃拟态 / 柔和渐变方向，但针对历史页做统一：

1. **信息层级更清楚**
   - 标题
   - 图表状态/趋势
   - 统计摘要
   - 告警记录

2. **统一组件语言**
   - 范围 tab 与指标 tab 圆角、描边、阴影一致
   - 统计区更接近信息卡片而不是纯文本
   - 低数据提示与空态提示使用统一胶囊样式

3. **提高可读性**
   - 深浅对比更明确
   - 避免白字叠浅玻璃背景
   - 避免图表主线被背景吞没

## 错误处理
本次不新增复杂错误系统，但保证：

1. 云端历史同步失败时，历史页仍可展示本地数据
2. 无数据场景有明确提示
3. 低数据场景有清晰说明，不伪装成正常完整趋势

## 测试与验证要求
### 手动验证重点
1. 切到 `1h` 时图表不再出现“全白看不到内容”
2. 无数据 / 少数据 / 正常数据三种状态可区分
3. 指标切换、时间切换时没有明显卡顿或错误点击映射
4. 历史页首次进入时能优先看到本地内容
5. UI 风格保持统一，更清晰、更协调

### 工程验证
- 运行 TypeScript 检查：`npm run type-check`
- 对修改文件执行诊断，确保无新增类型错误

## 不做的事
为了控制范围，本次不做：

1. 引入新的图表库
2. 重构整个 store 架构
3. 重做所有页面风格
4. 增加复杂缓存分层或后台任务系统

## 预期结果
完成后，历史页将具备以下效果：

- 1h 场景稳定可见，不再像“白屏”
- 页面进入与切换的主观速度更好
- 图表状态表达更清晰
- 整体视觉更统一、更协调、更像完整产品页面
